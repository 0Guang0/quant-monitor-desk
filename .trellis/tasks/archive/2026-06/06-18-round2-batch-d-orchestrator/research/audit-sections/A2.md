#### Important Issues

- **[orchestrator.py:82-123]** `emit_event` duplicates the `job_event_log` INSERT pattern already in `SyncJobStateMachine._transition_on_con` (`jobs.py:243-261`): same columns, UUID generation, `run_id` lookup, timestamp handling. ~25 lines are near-identical; the whole method is ~42 lines. **Fix:** Add `SyncJobStateMachine.emit_custom_event(...)` (or shared `_insert_job_event`) and delegate `DataSyncOrchestrator.emit_event` to it; delete duplicated SQL from orchestrator.

- **[orchestrator.py:69-77]** ResourceGuard block path updates `error_type`/`error_message` outside the state machine while status already moved via `transition(..., "FAILED_RETRYABLE")` (`orchestrator.py:63-68`). **Fix:** Extend `transition()` with optional `error_type` / `error_message` fields updated in `_transition_on_con` so guard-block handling stays in one module.

#### Suggestions

- **[orchestrator.py:343-393]** `run_reconcile(..., adapter=...)` never calls `adapter` — acceptable per MASTER §4.2 reconcile skeleton boundary, but add a one-line comment or `# noqa: adapter reserved for Round 3` to avoid reader confusion.

- **[orchestrator.py:125-260]** `run_incremental` is a long linear method (~135 lines). Not wrong for Batch D, but a private `_validate_and_write(...)` helper would improve testability without changing behavior.

- **[orchestrator.py:253]** `COMPLETED` transition runs in a new writer transaction after the validation/write `with` block closes (`orchestrator.py:167-252`). Consider passing `con` into final `transition` for atomicity with write, or document intentional split.

- **[sync/__init__.py:1]** Package docstring only — fine; no re-exports needed yet.

### Adversarial trigger result

**Found — mergeable block ≥20 lines:**

`DataSyncOrchestrator.emit_event` (`orchestrator.py:82-123`, **42 lines**) mirrors `SyncJobStateMachine._transition_on_con` event logging (`jobs.py:240-261`). Consolidating into `jobs.py` would delete ~25–30 lines from orchestrator and leave a thin delegate (~5 lines). This is the clearest ponytail violation: two writers for the same audit table with copy-pasted SQL.

**Secondary candidate (not primary):** `run_backfill` shard loop (`orchestrator.py:274-341`, **68 lines**) repeats guard-block / fetch-failure / transition patterns; could shrink ~20 lines with a `_run_shard_fetch(...)` helper, but behavior is not redundant—only structurally repetitive.

### What's Done Well

- Clean-table writes correctly funnel through `WriteManager` + `DbValidationGate` with `own_transaction=False` inside the orchestrator writer transaction (`orchestrator.py:229-241`).
- §6.5 fetch contract is followed on both FETCHING entry points; staging is never inserted by orchestrator SQL.
- `jobs.py` state machine is contract-driven: `_BASE_TRANSITIONS`, job-type overrides (`backfill` STAGED→PLANNED, `reconcile` PLANNED→WAITING_RECONCILE), and `InvalidTransitionError` guard invalid edges.
- `plan_backfill_shards` eco cap (`ECO_MAX_BACKFILL_DAYS_PER_TASK = 31`) is isolated and tested.
