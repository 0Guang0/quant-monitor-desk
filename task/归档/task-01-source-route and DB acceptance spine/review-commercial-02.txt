性能审查（只读 · ff605a85..HEAD）
范围：source route DB acceptance matrix（22 sources）、production_gate CI dry-run、SEC/Deribit port、acceptance_isolation
实测：本地 execute_documented_matrix(..., live_authorized=False) ≈ 4.1s（warm ~3.9s）；无 live fetch。

发现（按严重度，共 7 条）
1. [HIGH] 每矩阵行重复 bootstrap DuckDB + migration 校验
位置：backend/app/ops/source_route_db_acceptance.py:285-327（_bootstrap_acceptance_db），由 execute_documented_matrix 循环 22 次调用（source_route_db_acceptance_matrix.py:646-652）
问题：每行新建 ConnectionManager、抢写锁、apply_migrations()（含 15 个 migration 文件的 SHA256 校验）。22 行 ≈ 22 次完整 migration 扫描 + 22 次锁周期。
建议：在 execute_documented_matrix 入口 bootstrap 一次，把 cm 传入 spine.execute(..., cm=cm) 或矩阵级 context。
影响：potential impact — CI dry-run 主要 I/O 热点；矩阵扩源时线性变慢。
2. [HIGH] dry-run 仍每行写 route evidence（双写锁）
位置：backend/app/ops/source_route_db_acceptance.py:442-443（_persist_route_evidence）
问题：live_authorized=False 时虽早退 blocked，但已执行 preview_route_payload + DB INSERT。与 #1 叠加 ≈ 44 次 writer 锁周期/矩阵 run。
建议：dry-run / closure_mode=="dry_run" 时跳过 _persist_route_evidence；或矩阵结束批量 INSERT。
影响：potential impact — 多余 DuckDB 写入与锁竞争。
3. [HIGH] 每行 uncached 重载 registry YAML
位置：backend/app/ops/source_route_db_acceptance_matrix.py:334-356 → macro_incremental_common.py:424-434
问题：preview_route_payload 每行 SourceRegistry().load() + SourceCapabilityRegistry().load()（各读一次 YAML），22 行 ≈ 44 次磁盘读 + parse。
建议：矩阵 run 级缓存 (registry, caps, planner)；或 load_incremental_route_bundle 加 functools.lru_cache / 进程内单例。
影响：potential impact — CPU + 磁盘 I/O 随源数量线性增长。
4. [MEDIUM] closure 评估重复计算
位置：backend/app/ops/source_route_db_acceptance_matrix.py:671-680
问题：循环内已 evaluate_matrix_row_closure，summarize_matrix_closure 又对每行重算 + find_matrix_target_by_key 线性扫描（22×22 次比较）。
建议：summarize_matrix_closure 只聚合已有 closure_outcome；或去掉循环内预计算，只保留 summarize 一处。
影响：potential impact — 纯 CPU 冗余，源数增大后可见。
5. [MEDIUM] ensure_isolated_db 每次全量 registry sync
位置：backend/app/ops/acceptance_isolation.py:60-82
问题：DB 已存在仍 apply_migrations + SourceRegistry().load() + sync_to_db(tombstone_missing=True)。live acceptance harness（tier B/C）每次调用都走全路径。
建议：DB 已 migrate 且 registry 未变时 skip sync；或进程/路径级 memoize。
影响：potential impact — live matrix 路径（非 dry-run）显著拖慢。
6. [MEDIUM] SEC live fetch：无连接复用 + 高超时重试
位置：backend/app/datasources/fetch_ports/sec_edgar_port.py:198-224
问题：httpx.get() 每次新建连接；3 次重试 × timeout=60.0，失败时单 CIK 可阻塞 ≤180s；重试间无 backoff（可能打满 SEC rate limit）。
建议：模块级 httpx.Client 复用连接；重试加指数退避；acceptance 用更短 timeout（如 15s）。
影响：potential impact — live matrix / live acceptance 热路径阻塞风险（dry-run 不触发）。
7. [MEDIUM] SEC submissions 全量数组扫描
位置：backend/app/datasources/fetch_ports/sec_edgar_port.py:245-261
问题：accessionNumber 可达数百/上千条，循环到 max_filings 才 break；每行还拼 URL 字符串。
建议：先 zip/islice 限制到 max_filings 再构建 row dict。
影响：potential impact — live fetch CPU/内存；acceptance 仅取 3 条时浪费明显。
