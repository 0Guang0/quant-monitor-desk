## 2. SKIP 理由（§3.6 等价）

### 2.1 AUDIT.plan 冻结理由（复述 + 证实）

1. **无 hot path / SLA** — `DataHealthService.check_evidence_dir` 仅被单元测试与薄 CLI `data_health_cli.py` 调用；无 FastAPI 路由、无调度器、无 `production_equivalent_smoke.py` 路径。
2. **evidence-path 微型数据面** — 实现只读加载 ≤4 类 JSON manifest（raw/staging/validation/closeout）；v2 集成测指向 archive `execute-evidence`（数十个小文件，KB 级 manifest），**不扫描行情表/Parquet 分区**。
3. **任务卡 scope 排除重 I/O** — `R3Y_readonly_data_health_v1.md` §5.2：**不做 full market scan**、不写 DB、不 fetch、无 free SQL。
4. **零 DuckDB/Parquet/网络栈** — `data_health.py` 无 `duckdb`/`parquet`/`http` import；I/O 为 `Path.read_text` + `json.loads`。
5. **MASTER §10 无 perf 冻结行** — 通过条件仅为 `pytest`/`ruff` exit 0；无时长/内存数字可供 A6 PASS/FAIL。

### 2.2 实现路径 vs 规则函数

| 组件 | 证据路径是否调用 | 数据量级 |
| ---- | ---------------- | -------- |
| `load_evidence_bundle` / `_checks_from_bundle` | **是**（主路径） | manifest 条目数 ≈ staged pilot fetch 数（v2 caps 下个位数～数十） |
| `check_lineage_entry` / `check_staleness` | **是** | 每 manifest entry 一次；O(entries) |
| `check_daily_bars` / `check_metadata_rows` | **否**（仅单测直接调用） | 单测 1–3 行 fixture |
| `SourceRegistry.load()` | **是**（lineage 校验） | YAML 一次加载；`check_lineage_entry` 内可能重复 `load()` |

**结论：** 生产化入口当前仅为 **manifest 元数据健康**，非 domain 行级扫表；与 AUDIT.plan「无 full-market scan」一致。

## 3. 计划外 perf 风险扫描

> 按 `audit-adversarial-authority.md` A6：**即使 SKIP，仍须扫描** hot path、无界 I/O、批大小/内存尖峰。

### 3.1 Hot path

| 路径 | 是否 hot | 证据 | 风险 |
| ---- | -------- | ---- | ---- |
| `DataHealthService.check_evidence_dir` | **否** | 仅 `test_ops_data_health.py` + CLI | 无生产 SLA |
| `qmd data health --domain --db`（Batch 6） | **未实现** | `data_health_cli.md` §5 设计；v1 仅 `--evidence` | 未来须冻结窗口/行数 + ResourceGuard |
| R3Y staged pilot ingest | **未触及** | C-20 allowed 不含 `staged_pilot.py` | caps 在 R3Y 分支 |

**结论：** 本分支**不存在**可观测的生产 hot path；SKIP 合理。

### 3.2 无界 I/O

| 面 | 扫描 | 发现 | 评级 |
| -- | ---- | ---- | ---- |
| 网络 / live fetch | 代码审阅 + `rg` | 无；报告字段 `source_fetch_performed=False` 硬编码 | **无** |
| DuckDB / Parquet 全表扫 | 同上 | 零引用 | **无** |
| evidence JSON 读取 | `load_evidence_bundle` | 每 check 读 raw + 可选 staging/validation/closeout；**无 `max_bytes`/条目硬顶** | 见 NB-2 |
| manifest 列表遍历 | `_checks_from_bundle` | `manifest_entries` + `fetches` 双循环；规模受 staged pilot caps 约束，代码未 enforce | 见 NB-2 |
| bar/metadata 行级扫 | 主路径 | **未接入** `check_daily_bars`/`check_metadata_rows` | **当前无** |

**结论：** 无网络/DB 无界 I/O；JSON 读取在 staged 信任模型下可接受，但缺代码级体积上限。

### 3.3 批大小 / 内存尖峰

| 面 | 行为 | 当前量级 | 计划外风险 |
| -- | ---- | -------- | ---------- |
| JSON 全量 `read_text` | 整文件入内存 | KB 级 manifest | 巨型 manifest → 内存尖峰；无 ResourceGuard |
| `check_daily_bars` 去重 | `seen_keys` set + 行循环 | 单测 ≤3 行 | Batch 6 接 DB 窗口后须批处理/上限 |
| `SourceRegistry` | 每 `check_lineage_entry` 可触发 `load()` | 小 YAML | 多 entry 时重复 load；O(entries) 冗余 I/O |
| CLI subprocess | `test_dataHealthCli_readOnly_exitZero` 0.25s | 冷启动 Python | 非 BLOCKING；Batch 6 可考虑模块内测减 subprocess |

**结论：** 当前 evidence 规模下 CPU/内存**可忽略**；结构性风险在 Batch 6 扩展与 manifest 无硬顶。
