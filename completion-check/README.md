# completion-check · 核心资产

> 面向“完成声称”的七类检查语义核心。它可以被 skill、hook、workflow 或人工审计共同引用，但本身不负责触发、调度或执行。

## 定位

`completion-check` 只回答一个问题：

> 某对象声称“完成、可关账、可合并、可上线”时，这个声称能否扛住七类拷问？

本目录是消费端无关的核心资产：

- **不是 skill**：不定义模型何时自动调用，也不承载交互流程。
- **不是 hook**：不监听事件，不阻断命令。
- **不是 workflow**：不规定任务目录、角色文件名或阶段编排。
- **不是项目问题台账**：项目内复发模式与次数应放在各项目自己的 `completion-check.project.md`。

所有引用保持一层深度；七类正文只存在于 `REFERENCE.md`。

## 核心不变量

1. 七类 ID 固定为 `CC-1` 至 `CC-7`；英文代号是稳定机器键。
2. 每类只有一个权威拷问；解释、示例不得产生第二套问题文本。
3. 七类是**正交检查**，不是互斥标签；同一对象可以同时 FAIL 多类。
4. 每个对象固定输出七行；缺行、缺证据或未知状态都不得关账。
5. 检查判定只用 `PASS / FAIL / NA`；对象关账状态只用 `CLOSED / OPEN / BLOCKED`。
6. `CC-7 PASS` 只代表 defer 诚实，不代表对象已完成；对象仍应是 `BLOCKED`。
7. 运行事实优先于测试绿、文字声称、checkbox 或合成状态。
8. 项目示例、角色编排、文件命名与工具命令不得反向污染七类定义。

## 消费方式

- **Skill**：读取 `REFERENCE.md` 与 `CONTRACT.md`，组织证据收集、七行评估和解释。
- **Hook**：校验记录是否有七行、稳定 ID、合法状态和必填证据；不得凭关键词替代语义判断。
- **Workflow**：在阶段出口计算 `CLOSED / OPEN / BLOCKED`，并据此允许、阻断或转交。
- **人工审计**：独立复现证据，不复制执行者结论。

消费端可以增加交互和自动化，但不得复制或改写七类语义；需要改变语义时，只修改 `REFERENCE.md` 并同步版本与黄金样例。
