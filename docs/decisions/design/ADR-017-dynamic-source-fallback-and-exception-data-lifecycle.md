# ADR-017：动态数据源降级与异常数据生命周期

**状态：** Accepted（2026-07-11 经项目用户确认；本 ADR 与 `MIGRATION_MAP.md` 索引设计共同构成后续实现依据）

**日期：** 2026-07-11

**决策者：** 项目用户 / 产品负责人

**关联：** task-01-source-registry；ADR-0005（Primary / Validation / FallbackPolicy）；
ADR-016（来源矩阵的诚实关闸）

## 背景

Layer1–Layer5 的指标会持续新增、删减，数据源的授权、启用状态和可用性也会变化。若主数据源
全部不可用，监控不能因为没有数据而中断；但次数据源、失败原因和数据质量也不能被伪装成正常
主源数据。

当前设计已有 Primary / Validation / FallbackPolicy、`SourceRoutePlan`、`source_used`、
`quality_flags` 等基础，但尚未把动态启用、全领域次源优先级、质量异常连续监控、回补和旧版本
生命周期统一成一条可执行规则。特别是，可信 clean 表不得混入未验证数据，而用户仍需要在数据
质量异常时保持 Layer1–Layer5 的连续观察、计算和告警。

## 决策

### 1. 两种状态分离

数据源注册表是稳定的能力目录，定义来源、领域、角色、许可证、能力和初始准入；它不因一次
故障被运行时改写。

另设由项目管理员经受控配置/CLI 管理的**持久化启用覆盖层**。覆盖层只表达“当前是否允许这个
已登记来源参与某领域/操作的路由”，并须记录操作者、时间、原因、版本和撤销信息。未来可加
有权限的 UI，但不允许任务自行永久改开关。

### 2. 自动降级不是永久升格

每个数据领域维护固定的次源优先级。若所有 Primary 都失败，RoutePlan 自动按该顺序尝试所有
已登记、已审核、管理员启用、能力匹配的 Validation 来源。

这不把 Validation 升格为 Primary：实际被选中的次源必须以 `FallbackPolicy` 角色、
`degraded` 等级和明确的质量标记记录。未登记、禁用、能力不匹配或未经审核的来源绝不自动接入。

任何 Primary 失败原因都允许尝试合格次源以保证监控连续；若失败属于代码、适配器、格式或
schema 问题，还必须自动创建高优先级修复事件，记录失败来源、异常类型、影响领域和开始时间。

### 3. 来源风险与数据质量是独立标签

每条结果必须独立携带：

- **来源等级：** `PRIMARY` 或 `DEGRADED`；
- **质量等级：** `QUALITY_PASSED` 或 `QUALITY_FAILED`；
- **路由证据：** 实际 `source_used`、Primary 失败原因、RoutePlan、抓取/内容版本；
- **人工动作：** `MANUAL_REVIEW_REQUIRED`（适用时）。

因此，次源但质量通过是“降级可用”，质量失败则是“可连续观察但需要人工核对”；两者不得合并
成一个含混的“异常”标签。

### 4. 存储、读取与回补

```text
可信最终库（clean）
  已验证的主源历史与当前可信版本；不因主源恢复自动清理历史。

连续监控区（exception / degraded view）
  当前降级或质量异常数据；Layer1–Layer5 可以读取、计算、判断和告警，
  但前端与结论必须展示全部风险标签。

审计归档区（revision / audit archive）
  已被主源回补替代的降级/质量异常旧版本，以及 RoutePlan、质量报告和回补证据；
  不参与默认读取或默认回测。
```

主源恢复后，按领域自身频率重拉降级时段及前后缓冲窗口。验证通过后，主源版本成为默认读取的
可信版本；同一事实位置的旧异常活动版本迁入审计归档。不得把这项回补清理误解为删除可信最终
库中的正常历史数据。

质量异常数据可进入连续监控区并被 Layer1–Layer5 消费，但不得伪装为可信 clean 主值。若所有
来源均无可取得数据，必须返回 `MISSING`/失败状态，而不得伪造数值。

### 5. 归档与清理

仅对被回补替代的异常/降级完整 payload 执行按领域频率的归档保留；可信最终库的正常历史不由
本策略自动删除。

| 数据频率              | 完整异常 payload 最低保留期 |
| --------------------- | --------------------------: |
| 分钟、小时级          |                       90 天 |
| 日线、周线            |                        2 年 |
| 月度、季度            |                        5 年 |
| 年度、低频、事件/公告 |                        7 年 |

实际期限取 `max(上表基线, 相关指标最长计算窗口 + 2 个更新周期)`。来源、失败原因、RoutePlan、
质量标签、回补版本、内容 hash 与清理记录组成的精简审计索引长期保留。

清理任务必须先确认主源回补版本已验证写入、旧版本已成功归档、审计索引完整，才可删除到期
payload；禁止业务代码或人工临时直接删库。

## 备选方案

### 方案 A：直接修改源注册配置，不设启用覆盖层

优点是实现简单；缺点是频繁授权/停用会混淆稳定能力目录与运营状态，且难以审计和撤销。

**未选用。**

### 方案 B：任一主源失败即静默使用任一次源

优点是短期不断数；缺点是来源、质量和故障被隐藏，结果不可复盘，且容易把 Validation 伪装成
Primary。

**未选用。**

### 方案 C：将异常数据永久保留在可信最终库

优点是资料齐全；缺点是默认查询、回测和指标容易误读历史异常版本，可信库会逐渐变成混合库。

**未选用。**

### 方案 D：直接删除被回补的异常数据

优点是空间最少；缺点是无法验证回补是否正确、无法复盘当时的告警与路由决策。

**未选用。**

### 方案 E：可信最终库 + 连续监控区 + 审计归档区

保留可信默认读取，同时让监控在故障时不断线，并以归档控制异常数据的存储增长。

**选用。**

## 后果与模块责任

| 范围                 | 责任                                                                                    |
| -------------------- | --------------------------------------------------------------------------------------- |
| task-01              | Source Registry、Capability、管理员启用覆盖层、来源/质量标签契约；禁止调用方内存绕过。  |
| task-02              | 按领域次源顺序生成并持久化 RoutePlan、来源等级、失败原因和降级决策。                    |
| task-17 / task-18    | 调度和任务只能执行 RoutePlan，传递标签，触发回补/修复事件，不得自行换源。               |
| Layer1–Layer5 / 前端 | 连续监控区可消费；展示来源、降级、质量与人工核对标记；不得把异常结论伪装为正常结论。    |
| 数据质量 / 写入      | 维护可信 clean 与连续监控区的写入边界；回补、归档、清理经 WriteManager 和审计记录完成。 |

该决策增加了来源版本与数据标签的处理成本，也要求跨模块联调；换来的是故障期间连续监控、
可解释告警、可回补历史和干净的默认回测数据。
