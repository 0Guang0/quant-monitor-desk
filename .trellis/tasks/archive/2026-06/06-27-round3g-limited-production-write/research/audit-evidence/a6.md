## 3. 分级发现（P0–P3）

> 静态审阅项均标 **potential impact**；无 CWV 实测。

### P0 — BLOCKING

**无。**

### P1 — HIGH

**无。** 默认路径无无界网络、无调度热路径、无全市场扫描；单次 execute 仅 1 行 staging 写。

### P2 — MEDIUM（NON-BLOCKING under SKIP）

#### [P2] A6-P2-1 — approval `max_rows` 未贯穿 data-health 路径

- **Area:** Loading / Network（本地 CPU+I/O）
- **Location:** `limited_production_entry.py` → `rehearsal_runner._run_source_data_health`（`max_rows=1000` 硬编码，`limited_production_entry.py:428`）
- **Description:** `validate_approval_contract` 要求 `max_rows > 0` 且与 audit 对齐，但 promote runner 调用 DH 时 **未传入** `candidate.max_rows`；baostock profile 固定处理至多 1000 bars。
- **Impact:** potential impact — 若 approval 授权 `max_rows=100` 而证据目录含更大 `bars.json`，DH 仍可按 1000 行处理，**超出审批行数预算**的 CPU/内存；单次 pilot 量级小，不构成 hot path SLA 违约。
- **Recommendation:** 将 `_run_source_data_health` 改为接受 `max_rows: int` 参数，promote 路径传入 `candidate.max_rows`；与 `staged_pilot` / `fred_sandbox_pilot` 模式对齐。可在 post-pilot 硬化切片处理。

### P3 — LOW / 计划外 NON-BLOCKING

#### [P3] A6-P3-1 — dry_run 出口重复 key table 计数

- **Area:** Network（本地 DuckDB I/O）
- **Location:** `limited_production_entry.py:467-468`
- **Description:** `key_table_row_counts(production_db)` 对同一 DB 连续调用两次，结果赋给 `after_counts` / `before_counts`（before 未复用 earlier proof）。
- **Impact:** potential impact — 生产库越大、`KEY_TABLES` 越多，dry_run 退出时双倍 `COUNT(*)` 延迟。
- **Recommendation:** 复用 `before_proof` 阶段快照或单次调用后比较；pilot 频率低，可延后。

#### [P3] A6-P3-2 — target table 全表 COUNT 用于 before_proof

- **Area:** Network（本地 DuckDB I/O）
- **Location:** `limited_production_entry.py:_target_table_row_count`（`build_before_proof` 调用）
- **Description:** 对 `candidate.target_table` 执行无 `WHERE` 的 `COUNT(*)`；生产表行数增长时扫描成本线性上升。
- **Impact:** potential impact — GB 级生产表上 before_proof 变慢；活卡 §4.1 已选 `key_table_row_counts` 而非 `all_table_row_counts`，此为 **单表** 扫描，可接受于 pilot。
- **Recommendation:** 若未来频繁 promote，可改为按 approved key range 的有界计数（需 schema 支持）；当前 SKIP 不阻断。

#### [P3] A6-P3-3 — 证据 loader 无 row 硬顶（继承 R3G-01）

- **Area:** Loading
- **Location:** `rehearsal_loader.py` — `_load_baostock_bundle` / `_load_fred_bundle`
- **Description:** promote 复用 R3G-01 loader；`json.loads` 全量读入 bars/observations，**无** `R3G03_MAX_EVIDENCE_ROWS` 拒绝。
- **Impact:** potential impact — 恶意或误配 `evidence_dir` 可导致内存尖峰；默认 fixture 极小。
- **Recommendation:** live/execute 路径硬化时加 row 顶（≤ `candidate.max_rows` 或 contract 全局顶）；与 A6-P2-1 一并处理。

#### [P3] A6-P3-4 — live fetch CLI 标志未接线

- **Area:** Network
- **Location:** `PromoteRequest.allow_live_fetch` / `fred_authorization`；`data_commands.sandbox_clean_write_promote`
- **Description:** CLI 暴露参数；`run_limited_production_entry` **未读取**，无 `DataSourceService.fetch` 调用。
- **Impact:** potential impact **正面** — 当前零 live 网络风险；未来接线须挂载 RG + FRED caps + per-series row limit。
- **Recommendation:** 实现 live fetch 前对照 `fred_sandbox_pilot` 与 contract；本切片 SKIP 不受影响。
