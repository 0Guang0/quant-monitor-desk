### Findings

#### [MEDIUM] Fetch error messages persisted to `job_event_log` without redaction

- **Location:** `backend/app/sync/orchestrator.py` L154Γאף158, L307Γאף311; `backend/app/sync/jobs.py` L243Γאף261
- **Description:** `MASTER.plan.md` ┬º7 requires *πאלΘפשΦ»»µ╢טµב»Φה▒µץןµ▓┐τפ¿ Batch C `error_redaction`πאם*. `FetchLogWriter` and `WriteManager` call `redact_error_message()`, but `DataSyncOrchestrator` passes `fetch_result.error_message` directly into `SyncJobStateMachine.transition(..., message=...)`, which inserts the raw string into `job_event_log.message`. `orchestrator.py` and `jobs.py` do not import `error_redaction`.
- **Impact:** Adapter/auth failures that echo credentials (e.g. `AUTH_FAILED` with `api_key=...` or `Bearer ...` in the upstream error body) are stored in plaintext in `job_event_log`. Operators, future APIs, or DB backups could expose secrets that `fetch_log` already redacts.
- **Proof of concept:**
  1. Stub adapter returns `FetchResult(status="AUTH_FAILED", error_message="token=live-secret api_key=k123")`.
  2. `run_incremental` / `run_backfill` calls `self._jobs.transition(job_id, "FAILED_*", message=fetch_result.error_message)`.
  3. Query `SELECT message FROM job_event_log WHERE job_id = ?` Γזע contains `live-secret` and `k123`.
  4. Same fetch writes `fetch_log.error_message` with `[REDACTED]` (see `tests/test_data_adapter_contract.py::test_fetchLogWriter_redactsSensitiveErrorMessage`).
- **Recommendation:** Centralize redaction at the persistence boundary in `jobs.py`:

```python
from backend.app.util.error_redaction import redact_error_message

# In _transition_on_con, before INSERT:
safe_message = redact_error_message(message) or ""
```

Also redact in `orchestrator.begin_fetching` when updating `data_sync_job.error_message` (L73Γאף76) for consistency, and add `test_sync_orchestrator.py` coverage mirroring the fetch_log redaction test.

#### [LOW] `trigger_reason` not validated before `payload_json` persistence

- **Location:** `backend/app/sync/orchestrator.py` L275Γאף282
- **Description:** Backfill shards write `json.dumps({"trigger_reason": spec.trigger_reason or "eco_catchup"})` to `job_event_log.payload_json`. `MASTER.plan.md` ┬º6.3 documents six allowed trigger-reason values, but `SyncJobSpec` / `create_job` accept any string with no allowlist check.
- **Impact:** Low today Γאפ callers are internal (CLI/smoke/tests). A misconfigured caller could persist arbitrary strings (including accidental secrets) in `payload_json`. Not exploitable remotely without code execution.
- **Recommendation:** Validate against the contract enum at `create_job` or `run_backfill` entry; reject unknown values before `emit_event`.

### Positive Observations

- **Parameterized SQL throughout sync module.** Every `con.execute` in `orchestrator.py` and `jobs.py` uses `?` bind parameters for values; no f-string or `+` SQL concatenation with external input.
- **No secret material in sync source.** Adversarial grep found no hardcoded URLs, API keys, tokens, passwords, or bearer strings in `backend/app/sync/*`.
- **No Agent / free-SQL surface.** `backend/app/agents/__init__.py` is a one-line Round 4 placeholder; sync orchestration does not invoke agents or accept arbitrary SQL.
- **`job_event_log.payload_json` is minimal.** Only backfill shard events carry JSON, and the sole key is `trigger_reason` Γאפ no fetch params, headers, or connection strings.
- **`error_redaction.py` policy is sound.** Regexes cover `token`, `password`, `api_key`/`apikey`, `secret`, `authorization:`, and `Bearer` patterns; reused consistently in Batch C `fetch_log` and `write_manager`.
- **ResourceGuard messages are structured and non-sensitive.** `format_pause_event` emits decision/reason/metrics only; correctly routed to `job_event_log.message` with `RESOURCE_GUARD_PAUSED` prefix (tested).
- **Downstream table identifiers are allowlisted.** `clean_table` / `staging_table` flow into `WriteManager` and validators that call `quote_ident()` (`^[a-z][a-z0-9_]{0,62}$`) before dynamic `SELECT * FROM {quoted}` Γאפ outside sync files but on the orchestrator call path.
- **No stdout/logging in sync module.** No `print`, `logger`, or stderr emission that could leak credentials outside DB persistence.
