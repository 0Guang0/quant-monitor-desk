## Findings

### [A1-01] evidence 集成路径未执行日 K / metadata 规则 — **BLOCKING**

- **Location:** `backend/app/ops/data_health.py:565-636` (`_checks_from_bundle`)；`DataHealthService.check_evidence_dir`（`:678-704`）
- **Description:** 集成入口仅执行 lineage（`check_lineage_entry`）、staleness（`check_staleness`）与 staging manifest 缺失检查；**从未**调用 `check_daily_bars` / `check_metadata_rows`，也不读取 manifest 中的 `raw_file_paths` / `relative_paths` 载荷。
- **权威对照:** R3Y §3 目标 2–3；R3Y-DH-03/04/09；PROMPT_20 §6「daily bar checks」「cninfo metadata checks」「integration check against staged pilot evidence」；AC-DH-BIZ；MASTER Red Flag「CLI 空壳」。
- **Impact:** 对 staged pilot v2 evidence 的「数据健康」报告实质上仅为 manifest 血缘 + 陈旧性摘要；价量坏数据、metadata 缺口在真实 evidence 路径上不可见，与任务卡「不是 CLI 壳」矛盾。
- **证据:** v2 `raw_evidence_manifest_v2.json` 含 `fetches[].fetch_result.raw_file_paths` 与 `row_count: 2`，但 `_checks_from_bundle` 无 JSON 行加载逻辑；`test_dataHealthIntegration_v2Evidence_bundle` 仅断言 `overall_status` 非空与 gate 字段，不断言日 K 规则曾运行。
- **Recommendation:** 在 `_checks_from_bundle`（或只读 loader）中按 domain 解析 manifest 指向的 staged 文件，调用既有 `check_daily_bars` / `check_metadata_rows`（或委托 `DataQualityValidator.validate_rows`）；补 integration 负例（坏 bar fixture 经 service 路径 FAIL）。
- **Blocking:** **BLOCKING**

### [A1-02] 未复用 `DataQualityValidator` 契约 SSOT — **NON-BLOCKING**

- **Location:** `backend/app/ops/data_health.py` 全文；`research/integration-ledger.md` 行 `validators/data_quality.py`
- **Description:** 规则语义在 ops 层内联重实现，未 import / 委托 `DataQualityValidator.validate_rows`。
- **Impact:** 与 `data_quality_rules.yaml` 严重度、边界条件可能漂移；后续 Batch6 合并成本高。
- **Blocking:** **NON-BLOCKING**（MASTER §6.1 冻结子集已单测覆盖；属架构债）
