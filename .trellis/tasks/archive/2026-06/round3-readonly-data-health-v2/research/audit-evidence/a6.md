## 2. SKIP 理由（§3.6 等价 · 与 AUDIT.plan §1 A6 一致）

> Plan 原文：**「只读文件体检无 hot path SLA」** — 本任务为 evidence 目录只读校验，无生产调度/API 热路径，无冻结 SLA。

### 2.1 五条证实

1. **无 hot path / SLA** — `DataHealthService.check_evidence_dir` 仅被单元测试与 `data_health_cli.py` 调用；无 FastAPI 路由、无调度器、无 `production_equivalent_smoke.py` 挂载点。
2. **evidence-path 微型数据面** — v2 在 v1 manifest 检查之上增加 5 个 profile（whitelist / FRED / TDX / staged_pilot_v3 / rollup）；均通过 `_read_profile_json` 读单文件 JSON 或 bounded glob，**不扫描行情表/Parquet 分区**。
3. **任务卡 scope 排除重 I/O** — `R3E` §7–§8：No full data root scan；No fetch；No DB write；Red Flags 禁止 network fetch 与 full scan。
4. **零 DuckDB/Parquet/网络栈** — `data_health.py` 无 `duckdb`/`parquet`/`httpx`/`requests` import；I/O 为 `Path.read_text` + `json.loads` + 有限 `glob`。
5. **MASTER §10 无 perf 冻结行** — 通过条件仅为 pytest/ruff/compileall exit 0；无时长/内存数字可供 A6 PASS/FAIL。

### 2.2 实现路径 vs 规则函数

| 组件                                            | v2 主路径是否调用      | 数据量级                                                       |
| ----------------------------------------------- | ---------------------- | -------------------------------------------------------------- |
| `_checks_model_input_whitelist`                 | **是**（profile）      | 0–N 个 YAML；`specs/model_inputs` 未合并时 BLOCKED             |
| `_checks_fred_sandbox`                          | **是**                 | 单 JSON `fred_evidence.json`                                   |
| `_checks_tdx_manual_probe`                      | **是**                 | 单 JSON `tdx_probe_evidence.json`                              |
| `_checks_staged_pilot_v3`                       | **是**                 | `pilot_v3_manifest.json` + manifest fetches 列表               |
| `_checks_source_readiness_rollup`               | **是**                 | rollup 内 **O(profiles)** 子目录递归；由 manifest 声明 bounded |
| `check_daily_bars` / `check_metadata_rows`      | **否**（单测直接调用） | fixture 1–3 行                                                 |
| v1 `load_evidence_bundle` / staged_pilot_bundle | **是**（默认 profile） | ≤4 类 JSON manifest                                            |

**结论：** 生产化入口为 **profile 化 evidence 元数据体检**，非 domain 行级扫表；与 AUDIT.plan「无 hot path SLA」一致。

## 3. 计划外 perf 风险扫描

> 按 `audit-adversarial-authority.md` A6：**即使 SKIP，仍须扫描** hot path、无界 I/O、批大小/内存尖峰。

### 3.1 Hot path

| 路径                                          | 是否 hot   | 证据                                | 风险                                 |
| --------------------------------------------- | ---------- | ----------------------------------- | ------------------------------------ |
| `check_evidence_dir` + v2 profiles            | **否**     | 测试 + CLI only                     | 无生产 SLA                           |
| `source_readiness_rollup` 子 profile 递归     | **否**     | manifest 驱动；深度 = rollup 条目数 | 恶意 rollup 可放大调用次数 → 见 NB-2 |
| `qmd data health --domain --db`（未来 Batch） | **未实现** | v2 仅 evidence profile              | 未来须冻结窗口 + ResourceGuard       |

**结论：** 本分支**不存在**可观测的生产 hot path；SKIP 合理。

### 3.2 无界 I/O

| 面                            | 扫描                              | 发现                                                         | 评级       |
| ----------------------------- | --------------------------------- | ------------------------------------------------------------ | ---------- |
| 网络 / live fetch             | 代码审阅                          | 无；`source_fetch_performed=False` 硬编码                    | **无**     |
| DuckDB / Parquet 全表扫       | 同上                              | 零引用                                                       | **无**     |
| full data root scan           | R3E §7 + 实现                     | `evidence_dir_within_project` 约束 CLI；profile 读显式文件名 | **无**     |
| JSON 全量 `read_text`         | `_read_json`                      | **无 `max_bytes`/条目硬顶**                                  | 见 NB-1    |
| rollup 子目录遍历             | `_checks_source_readiness_rollup` | profiles 键数未代码 enforce 上限                             | 见 NB-2    |
| whitelist `glob("**/*.yaml")` | `_checks_model_input_whitelist`   | 限于 `evidence_dir`；非全仓库                                | **可接受** |

### 3.3 批大小 / 内存尖峰

| 面                      | 行为                               | 当前量级                    | 计划外风险                                                    |
| ----------------------- | ---------------------------------- | --------------------------- | ------------------------------------------------------------- |
| JSON 全量入内存         | `_read_json`                       | KB 级 fixture               | 巨型 JSON → 内存尖峰；与 R3E §7「大文件应摘要化」存在表述差距 |
| rollup 嵌套 checks      | `checks.extend(sub_report.checks)` | 测试 fixture 个位数 profile | 多 profile + 深嵌套 → 报告列表膨胀                            |
| `SourceRegistry.load()` | lineage 路径                       | 小 YAML                     | 多 entry 重复 load（v1 遗留，v2 主路径较少触及）              |
| CLI subprocess          | 最慢 0.35s                         | 冷启动 Python               | 非 BLOCKING                                                   |

**结论：** 当前 evidence 规模下 CPU/内存**可忽略**；结构性风险在 rollup 无顶与 JSON 无硬顶。
