### Findings

#### [P1] `target_table` 读路径绕过 `quote_ident` — SQL 注入面

- **Location:** `backend/app/ops/sandbox_clean_write/limited_production_entry.py:139`, `:370`
- **BLOCKING:** YES（合并前修复）
- **Description:** `_target_table_row_count` 与 `_production_clean_write` 末尾 COUNT 使用 `f'SELECT COUNT(*) FROM "{table}"'`，未调用 `backend.app.db.sql_identifiers.quote_ident`（`^[a-z][a-z0-9_]{0,62}$`）。同一模块内 `_table_schema_hash` 对表名使用参数化 `?`，WriteManager 写路径亦经 `quote_ident` 校验。
- **Impact:** 恶意或受损的 approval YAML 可在 `target_table` 注入 DuckDB SQL（例如含 `"` 跳出标识符），在 promote 读证明阶段执行非预期查询；写路径虽会被 `WriteManager` 拒绝非法标识符，但读路径仍可在写门禁之前泄露或破坏数据。
- **Proof of concept:**
  ```yaml
  # approval 中 target_table（audit_decision 同字段对齐即可通过 equality）
  target_table: 'market_bar_clean"; SELECT * FROM write_audit_log; --'
  ```
  触发 `build_before_proof` / dry_run 后 COUNT 时执行拼接 SQL。
- **Evidence:**
  ```bash
  rg -n 'execute\(f.*SELECT COUNT' backend/app/ops/sandbox_clean_write/limited_production_entry.py
  # 139: return int(con.execute(f'SELECT COUNT(*) FROM "{table}"').fetchone()[0])
  # 370: clean_count = con.execute(f'SELECT COUNT(*) FROM "{candidate.target_table}"').fetchone()[0]
  ```
- **Recommendation:** 所有动态表名统一 `quote_ident(candidate.target_table)`；在 `validate_approval_contract` 或 `ApprovalCandidate.from_dict` 对 `target_table` 做与 `quote_ident` 相同的 regex 校验（fail-closed）。

#### [P2] CLI `--allow-live-fetch` / `--fred-authorization` 在 promote 路径未生效

- **Location:** `backend/app/cli/main.py:96-98`, `backend/app/ops/sandbox_clean_write/limited_production_entry.py:80-81,421-426`
- **BLOCKING:** NO
- **Description:** 活卡 §9.4 要求 live fetch 仅在 `live_fetch_authorized: true` 且 FRED authorization artifact 存在时允许；CLI 暴露 `--allow-live-fetch` 与 `--fred-authorization`，`PromoteRequest` 携带二者，但 `run_limited_production_entry` 未调用 `validate_fred_authorization`，也未将旗标传入 `DataSourceService` / `load_rehearsal_bundle`。
- **Impact:** 当前默认行为偏安全（fixture dry_run、无 live 拉取），但操作者可能误以为已授权 live fetch；未来接线时若忘记门禁易产生静默误配。
- **Recommendation:** 要么删除未接线 CLI 旗标直至实现；要么在 promote 内对 `fred` 源强制 `validate_fred_authorization` + `candidate.live_fetch_authorized` 双重检查后再传 `DataSourceService`。

#### [P2] CLI fail-closed 测试未覆盖 audit / before_proof 缺失

- **Location:** `backend/app/cli/data_commands.py:365-376`, `tests/test_round3g_limited_production_clean_write.py`（仅有 approval、rollback CLI 测）
- **BLOCKING:** NO
- **Description:** CLI 层已对 `audit_decision`、`before_proof` 做 `is_file()` 拒绝（与 approval、rollback 对称），但 §10.1 `PromoteCli` 矩阵仅测 `missing_approval_file` / `missing_rollback_plan`。
- **Impact:** 回归时可能意外删除 audit/before CLI 检查而不被测试捕获；运行时仍 fail-closed，风险为防护退化。
- **Recommendation:** 增补 `test_PromoteCli_missingAuditDecision_blocks` / `test_PromoteCli_missingBeforeProof_blocks`。

#### [P3] `production_db_path` 自动 `mkdir` 扩大可写目录面

- **Location:** `backend/app/ops/sandbox_clean_write/limited_production_entry.py:97-98`, `:334`
- **BLOCKING:** NO
- **Description:** `_assert_production_db_allowed` 在 DATA_ROOT / `.audit-sandbox` 校验通过后会 `mkdir(parents=True)`。
- **Impact:** 审批路径在允许根下可创建任意子目录；需配合 approval 人工门控，属纵深防御缺口。
- **Recommendation:** execute 路径要求 DB 文件或父目录已存在，或记录审计事件。
