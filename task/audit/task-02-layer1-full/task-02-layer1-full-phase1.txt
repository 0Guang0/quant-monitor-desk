──────── Project Summary ────────

### Top Issues

1. **Partial failures are repeatedly hidden by returning only the last result.**
   Multiple execution paths aggregate multiple bindings or shards but surface only the final outcome, so earlier failures or skipped shards can be masked by a later success. This can make failed or partially completed syncs look successful in audit and retry flows. Representative paths: `backend/app/sync/scheduler.py`, `backend/app/sync/binding_executor.py`.

2. **Job state transitions are not concurrency-safe.**
   Several paths validate state based on an earlier read, then update by `job_id` only or perform the transition after releasing the guarded connection. Competing workers can both pass validation, overwrite each other’s state, or log misleading events. Representative paths: `backend/app/sync/jobs.py`, `backend/app/sync/orchestrator.py`.

3. **Resource guard and blocked-run handling leave inconsistent job records.**
   Guard-blocked execution can return retryable failure without persisting the matching job transition/event, while dry runs are blocked even though they consume no resources. This weakens operator planning and can leave jobs stuck in `PLANNED`. Representative paths: `backend/app/sync/runners.py`, `backend/app/sync/scheduler.py`, `backend/app/sync/orchestrator.py`.

4. **External input validation is too loose across registry, scheduler, and facade boundaries.**
   YAML/profile/API-adjacent inputs accept malformed or ambiguous values, including string booleans treated as `True`, non-string formulas, duplicate indicator IDs, invalid feature outputs, missing job types, invalid job types, and non-positive fixture row counts. These issues can silently corrupt execution semantics or fail later with unclear errors. Representative paths: `backend/app/sync/indicator_binding.py`, `backend/app/sync/scheduler.py`, `backend/app/sync/layer1_sync_facade.py`.

5. **Mapper robustness is weak for malformed external macro data.**
   Bad timestamps, non-normalized dates, and non-numeric values can abort an entire evidence bundle. The mapper also hard-codes unit/frequency metadata and generates per-row timestamps, which can corrupt staging semantics and reduce deterministic behavior. Path: `backend/app/sync/mappers/macro_series.py`.

6. **SQL and table-name handling creates injection or invalid-SQL risk.**
   Raw table identifiers are interpolated directly into staging/reconcile SQL, with values originating from fetch implementations or caller-controlled conflict data. This needs identifier validation/quoting or a constrained allowlist. Path: `backend/app/sync/runners.py`.

7. **Watermark and date planning have correctness and scalability issues.**
   Invalid `max_days` can cause infinite or backwards loops, cold-start lookback returns one extra inclusive day, and per-series watermark queries scale poorly for large macro batches. Representative paths: `backend/app/sync/jobs.py`, `backend/app/sync/watermark.py`.

8. **Recovery and idempotency decisions rely on insufficient evidence.**
   Existing `job_id` reuse is treated as idempotent without confirming the stored spec matches the requested spec, and recovery can mark a `WRITING` job as `COMPLETED` based only on `write_id`. These can hide conflicting runs or partial writes. Representative paths: `backend/app/sync/jobs.py`, `backend/app/sync/orchestrator.py`.

9. **Domain-specific execution assumptions can route jobs into the wrong schema or semantics.**
   `full_load` treats non-macro domains as bar-like domains if they have clean-write targets, and omitted `instrument_id` can reuse a registry watermark key as a fetch identifier depending on domain. This risks invalid dispatch for disclosures, crypto, or other non-bar domains. Path: `backend/app/sync/binding_executor.py`.

10. **Global monkey patching is unsafe under concurrent execution.**
   `macro_incremental_validation_patch()` mutates `_PipelineMixin._validate_staging` at class level, so concurrent non-macro runs can observe macro-specific validation behavior. Path: `backend/app/sync/binding_executor.py`.

### Module Hotspots

- `backend/app/sync/binding_executor.py`: High-severity execution correctness issues: global monkey patching, partial shard result masking, domain misclassification, and ambiguous watermark/instrument semantics.
- `backend/app/sync/runners.py`: Operational consistency and security hotspot: guard-blocked jobs do not persist state, raw identifiers are interpolated into SQL, successful shard writes omit job-level `write_id`, and reconcile validation appears to compare against the wrong source rows.
- `backend/app/sync/jobs.py`: Core job lifecycle risks: infinite loop on invalid shard sizing, weak idempotency checks, race-prone status transitions, and import-time contract loading failures.
- `backend/app/sync/orchestrator.py`: State coordination and recovery weaknesses: check/transition race, unsafe write recovery, dropped keyword arguments, and a broken public type annotation under runtime type-hint evaluation.
- `backend/app/sync/indicator_binding.py`: Registry validation is under-specified, allowing type coercion, duplicate IDs, and invalid feature output definitions.
- `backend/app/sync/mappers/macro_series.py`: Concentrated external-data robustness problems in timestamp/date/value parsing and metadata derivation.

### Cross-Cutting Concerns

- **Inconsistent aggregation of multi-step outcomes.**
  Multi-binding and multi-shard flows collapse results to the last item instead of preserving partial failures or skipped shards. Seen in `backend/app/sync/scheduler.py` and `backend/app/sync/binding_executor.py`.

- **Weak boundary validation and fail-late behavior.**
  Inputs from YAML, API-like facades, job specs, and kwargs are accepted without strict validation, then fail later or silently change behavior. Seen in `backend/app/sync/indicator_binding.py`, `backend/app/sync/scheduler.py`, `backend/app/sync/layer1_sync_facade.py`, and `backend/app/sync/orchestrator.py`.

- **Job lifecycle state can diverge from returned results.**
  Several paths return success/failure/skipped statuses that do not match persisted `data_sync_job` state or metadata. Seen in `backend/app/sync/runners.py`, `backend/app/sync/binding_executor.py`, `backend/app/sync/scheduler.py`, and `backend/app/sync/orchestrator.py`.

- **Concurrency assumptions are too optimistic.**
  Class-level mutation, read-then-write transitions, and check-then-transition gaps create race windows under concurrent sync execution. Seen in `backend/app/sync/binding_executor.py`, `backend/app/sync/jobs.py`, and `backend/app/sync/orchestrator.py`.

- **External-data parsing lacks explicit bad-record policy.**
  Mapper code often lets one malformed field abort the whole batch instead of skipping, quarantining, or reporting row-level errors. Seen most clearly in `backend/app/sync/mappers/macro_series.py`.

- **Audit and lineage metadata are incomplete or misleading.**
  Returned scheduler metadata can disagree with actual defaults used for execution, successful shard writes may not persist `write_id`, and recovery may overstate completion. Seen in `backend/app/sync/scheduler.py`, `backend/app/sync/runners.py`, and `backend/app/sync/orchestrator.py`.

- Reject unknown `JobType` values and non-positive `row_count` in `backend/app/sync/layer1_sync_facade.py`.
- Capture one `created_at` timestamp per macro bundle and wrap date/value parsing with explicit skip/report behavior in `backend/app/sync/mappers/macro_series.py`.
- Persist a blocked transition/event when `ResourceGuard` prevents execution in `backend/app/sync/runners.py`.
- Replace raw SQL identifier interpolation in `backend/app/sync/runners.py` with identifier validation/quoting or an allowlist.
- Remove or reject unused `**kwargs` in `backend/app/sync/orchestrator.py` so caller typos fail fast.
- Import `DataSourceService` where it is used in public annotations in `backend/app/sync/orchestrator.py`.
