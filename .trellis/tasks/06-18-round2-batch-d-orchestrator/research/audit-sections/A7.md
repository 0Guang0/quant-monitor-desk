## 1. ResourceGuard before FETCHING

### Requirement

`AUDIT.plan.md` §A7 · `MASTER.plan.md` §4.4 · AC-5:

- Every path entering `FETCHING` must call `ResourceGuard(con=writer_con).check()` first.
- Backfill shards re-enter `PLANNED` between tasks; each shard must re-check before its own `FETCHING`.

### Evidence

**Single choke point.** `DataSyncOrchestrator.begin_fetching()` is the only code that transitions a job to `FETCHING`:

```42:80:backend/app/sync/orchestrator.py
    def begin_fetching(self, job_id: str) -> bool:
        """Enter FETCHING after ResourceGuard.check(); return False if blocked."""
        ...
            guard = ResourceGuard(con=con)
            decision, reason = guard.check()
            if decision in (Decision.PAUSE, Decision.HARD_STOP):
                ...
        ...
        self._jobs.transition(job_id, "FETCHING")
        return True
```

**Call sites.** `grep` over `backend/app/sync/` shows `begin_fetching` is invoked from:

| Path | Location | Guard before fetch |
|------|----------|-------------------|
| `run_incremental` | `orchestrator.py` L138 | Yes |
| `run_backfill` (per shard) | `orchestrator.py` L284 | Yes |

No other `transition(..., "FETCHING")` exists outside `begin_fetching`. `run_reconcile` does not enter `FETCHING` (reconcile delegate only) — consistent with MASTER §4.2 skeleton boundary.

**Writer connection.** Guard uses `ConnectionManager.writer()` inside `begin_fetching`, matching DECISIONS §1 single-writer policy.

**Tests.**

4. Query DB:
   - `data_sync_job.status == "FAILED_RETRYABLE"` ✓
   - `status != "RESOURCE_GUARD_PAUSED"` ✓
   - `job_event_log.message` contains `RESOURCE_GUARD_PAUSED` ✓ (from `format_pause_event`)

