### Important Issues

1. **[NON-BLOCKING] loader 错误模型缺少负向测试**
   - **位置：** `backend/app/datasources/provider_catalog.py:23-30`；`tests/test_provider_catalog.py`（缺失）
   - **描述：** `load_provider_catalog` 在项目根外路径抛出 `ProviderCatalogError`，对非 dict 根、`providers` 非 list 亦抛错；测试仅覆盖 happy path（`test_loadProviderCatalog_returnsProviders`）。与同仓库 `test_source_registry.py` / `test_raw_store.py` 的 outside-root 模式不一致。
   - **建议：** 增加 `test_loadProviderCatalog_outsideRoot_raises` 与 `test_loadProviderCatalog_invalidShape_raises`（`pytest.raises(ProviderCatalogError)`）。
   - **阻塞：** NON-BLOCKING

2. **[NON-BLOCKING] design 与 catalog 的 yahoo_finance status 漂移**
   - **位置：** `specs/datasource_registry/provider_catalog.yaml`（yahoo_finance `status: active`）；`design.md` §1.2（初值 `proposed_disabled_source`）
   - **描述：** 设计表写 yahoo 为 `proposed_disabled_source`，实现为 `active`。`source_capabilities.yaml` 对 yahoo 无显式 `status`，故 `test_catalogStatus_matchesCapabilityRegistry` 不拦截。posture（`enabled_by_default: false`、`validation_only: true`、在 `PROPOSED_EXTERNAL_SOURCE_IDS` 中受 production 禁用测试）仍正确。
   - **建议：** 统一 design 或 catalog 其一；若保持 `active`，在 design §1.2 注明与 capability 无 status 字段时的约定。
   - **阻塞：** NON-BLOCKING

### Suggestions

1. **`provider_catalog.py:38-39`** — `provider_for_source` 对非 dict entry 静默 `continue`；若 YAML 手工编辑引入坏行，返回 None 而非 `ProviderCatalogError`，调试成本略高。可在 loader 或独立校验函数 fail-fast（R3G 前可选）。

2. **`tests/test_provider_catalog.py`** — 可增加全量断言 `runtime_source_copy_allowed is False`（当前仅 openbb 单测 + YAML 人工核对）；防止未来条目误标。

3. **`tests/test_provider_catalog.py`** — 可增加 per-source `source_type`/`license_type` 与 `source_registry.yaml` 交叉相等测试（对抗性脚本核对：当前 25 源零漂移，但无测试锁死）。

4. **`service.py:325`** — `provider_metadata_only` → `describe_architecture_reference` 为 registry `domain_roles` 配套最小映射，合理；若严格遵循 design §3「不在 fetch 热路径调用 catalog」，可在注释标明「domain default only; catalog loader 未接入」以免后续误接。
