# completion-check · 七类七拷问

> **SSOT：** 七类定义、稳定 ID、英文代号与权威拷问只在本文件。

## 1. 评估对象

`completion-check` 评估的不是“代码好不好”，而是一个明确的**完成声称**：

- **对象**：一个 AC、Slice、任务、交付物、变更集或可独立关账的能力。
- **声称**：完成、可关账、可合并、可上线、审计 PASS 等。
- **权威**：该对象必须满足的计划、设计、契约、决议或用户明确要求。
- **运行事实**：由正式入口、报告、数据库、日志、文件产出或可复现命令证明的观察结果。

没有明确对象、声称和权威时，只能探索，不能关账。

## 2. 两种用法不得混淆

### 七行评估

七类是七个**正交检查面**。每个对象必须逐类回答；同一事实可以导致多类同时 FAIL。

### 问题归档

项目台账若需要给某个发现指定“主分类”，使用后文的边界规则选择最直接的根因归属。主分类只用于归档，不得据此跳过其余六项检查。

## 3. 判定与关账

### 检查判定

| 判定   | 含义                         | 最低要求                                 |
| ------ | ---------------------------- | ---------------------------------------- |
| `PASS` | 该类拷问得到可复现的肯定答案 | 产出与证据均非空                         |
| `FAIL` | 已观察到违反该类要求的事实   | 写明缺口、影响与复现证据                 |
| `NA`   | 该检查面对本对象客观不存在   | 写明为何不存在；不得用来掩盖未知或未检查 |

### 对象状态

| 对象状态  | 条件                                                                      |
| --------- | ------------------------------------------------------------------------- |
| `CLOSED`  | 七行齐全，无 `FAIL`，证据完整，且不存在有效 blocker/defer                 |
| `OPEN`    | 任一行 `FAIL`、缺行、缺证据、状态未知，或仍有可完成但未完成事项           |
| `BLOCKED` | 七行无 `FAIL`，但对象存在经 `CC-7` 验证为诚实且可追踪的客观 blocker/defer |

`CC-7 PASS` 证明“没有假装完成”，不等于对象完成。

## 4. 七类总表

| ID     | 分类（代号）              | 核心判断               | 唯一权威拷问                                                                                                                     |
| ------ | ------------------------- | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `CC-1` | 假绿 `false-green`        | 绿是否真能证明业务     | 是否存在至少一条走正式入口、以业务 outcome 为断言的核心路径证据，且删除测试噪音后结论仍成立？                                    |
| `CC-2` | 半成品 `shell-done`       | 权威能力有没有真实落地 | 权威承诺的每项能力是否已在生产路径真实落地，而非仅有函数、stub、mock、状态或 synthetic PASS？                                    |
| `CC-3` | 入口分裂 `entry-split`    | 正门与其他入口是否同规 | 用户正式入口与测试或其他正式入口是否使用同一接线与规则，并对同参给出一致结论？                                                   |
| `CC-4` | 档位膨胀 `mode-inflation` | 彩排是否冒充开业       | 证据所处运行档位是否与关账声称及产出字段完全一致？                                                                               |
| `CC-5` | 交付偏航 `delivery-drift` | 实际形是否对齐权威形   | 实际交付的步骤链、契约字段和产出形状是否与权威一致；任何偏差是否已有审阅后的书面变更？                                           |
| `CC-6` | 卫生债 `hygiene-debt`     | 能清的债是否继续堆     | 本轮能够消除的绕路、重复和临时补丁是否已修复，或已登记 owner、任务 ID 与删除条件？                                               |
| `CC-7` | 诚实延期 `honest-defer`   | 做不了时是否诚实关闸   | 客观无法完成的部分是否保持 `BLOCKED / FAIL_EXTERNAL / DEFERRED`，有依据与可追踪 ID，且未被 mock、replay 或 synthetic PASS 掩盖？ |

## CC-1 · 假绿 false-green

**一句话：** 套件全绿，但绿的不是业务证明。

**通过标准**

- 至少一条核心路径从用户正式入口进入。
- 断言落在可感知 outcome，例如数据库状态、报告字段、生成文件、业务状态迁移或外部可观察结果。
- 去掉只测调用次数、非空、无异常、文档存在等噪音测试，关账结论仍成立。
- 测试没有通过缩小目标、放宽断言或测试专属替身来制造绿灯。

**典型 FAIL**

- 只断言 `True`、无异常、调用次数或结果非空。
- 测文件、枚举、catalog、文档措辞，不运行生产行为。
- 集成测试依赖 monkeypatch/autouse 注入，正式入口没有同等待遇。
- 为过关删除失败分支、放宽精确断言或让测试永不进入生产路径。

**证据最低要求**

正式入口复现命令或调用方式；核心路径测试全名；被验证的业务 outcome；必要时给出“测试注入路径 vs 正式入口”的对比。

**NA 边界**

只有当对象纯粹是无运行行为的静态文本资产，且声称不包含可执行效果时才可 `NA`。

---

## CC-2 · 半成品 shell-done

**一句话：** 权威要的能力还没有真正实现。

**通过标准**

- 将权威中的每项能力逐项映射到真实生产行为。
- 实现覆盖完整链路，而非只有接口、函数名、状态转换、占位报告或测试替身。
- 权威要求的持久化、迁移、双写、回读、触发或失败处理均真实存在。

**典型 FAIL**

- runner、handler 或 command 只有壳，缺少核心业务步骤。
- 子任务没有真实验收产出，父流程手工合成 PASS。
- 决议或 ADR 已关闭，但实现仍待后续。
- 只完成多路落盘中的一侧，或机制在库内可测但正式入口未接通。

**证据最低要求**

权威条目到实现/产出的逐项映射；正式入口运行事实；不得仅提供“有函数”“有测试”“状态为 COMPLETED”。

**NA 边界**

完成声称必然涉及“有没有做”，通常不得 `NA`。

---

## CC-3 · 入口分裂 entry-split

**一句话：** 同一能力，正门和后门不是一套规矩。

**通过标准**

- 明确用户实际会走的正式入口。
- 测试入口、CLI、API、scheduler、批处理或其他正式入口复用同一配置、依赖接线与判定规则。
- 同参、同档位、同依赖条件下，各入口给出一致结论和等价产出。

**典型 FAIL**

- 测试注入依赖后通过，正式入口裸奔失败。
- 一个正式入口临时启用能力，另一个仍读取默认禁用配置。
- dry-run/mock 路径被覆盖，live 真链接缝未覆盖。
- 旧验收 helper 与权威验收入口并存且结论不同。

**证据最低要求**

正式入口名称与复现；至少一个同参入口对比；依赖、配置和输出差异的定位。

**NA 边界**

只有当对象确实只有一个入口，且不存在测试专属接线或替代执行路径时可 `NA`。

---

## CC-4 · 档位膨胀 mode-inflation

**一句话：** 彩排不能冒充开业。

**通过标准**

- 明确声称的运行档位，例如 live、replay、dry-run、sandbox、stub。
- 报告字段、运行配置、数据源、标识符与证据均指向同一档位。
- 低档证据只支持对应低档结论，不被升格为高档授权。

**典型 FAIL**

- replay/fixture 运行被写成 live-ready。
- 报告标 live，但 fetch/run ID 含 mock、stub、replay 痕迹。
- sandbox DONE 被当作生产主路径闭环。
- dry-run PASS 被用于生产授权。

**证据最低要求**

声称档位；实际运行参数；产出中的档位字段；能识别数据来源或执行模式的证据。

**NA 边界**

仅当对象完全不存在运行档位差异时可 `NA`；“未记录档位”是 `FAIL` 或未知，不是 `NA`。

---

## CC-5 · 交付偏航 delivery-drift

**一句话：** 能跑，但不是权威要的那条路。

**通过标准**

- 权威要求的步骤链、字段、结构、边界和产出形状均与实际一致。
- 所有必填字段有真实值，不以空值、占位符或拼接字符串充数。
- 任何偏差都先形成经审阅的 plan/note/ADR/用户决议，再据此实现。

**典型 FAIL**

- 完整金路径被缩成短链，后续校验、持久化或触发步骤消失。
- 报告已 PASS，但契约必填字段缺失或为空。
- 多入口产生不同的信封、closure 或字段形状。
- checkbox 已勾，实际仍未满足 AC 回读清单。

**证据最低要求**

权威引用；实际步骤/字段/形状；逐项差异；若有偏差，给出已审阅的书面变更引用。

**NA 边界**

完成声称通常都有预期形，通常不得 `NA`。

---

## CC-6 · 卫生债 hygiene-debt

**一句话：** 本轮能做却继续留下的绕路，会叠成债。

**通过标准**

- 本轮新增或新发现的绕路、重复、临时开关和调用方补丁已在共享根因处修复。
- 不能同步修复但确需保留的债，有 owner、任务 ID、影响范围和可验证删除条件。
- 不用“本关不阻塞”“后续再说”替代登记。

**典型 FAIL**

- 用 `sys.path`、临时环境变量或全局 patch 绕过正式接线。
- 同一配置、枚举或初始化逻辑维护多份 SSOT。
- 各调用方分别打补丁，公共根因仍在。
- 本轮能修却假称阶段外事项，且无任务 ID 或删除条件。

**证据最低要求**

本轮触及的相关接缝；已清理的事实，或债务条目的 owner、任务 ID、删除条件和台账位置。

**NA 边界**

对象未触及相关接缝、未新增绕路、也未新发现可行动卫生债时可 `NA`。

---

## CC-7 · 诚实延期 honest-defer

**一句话：** 客观做不了时诚实关闸，不用替身补绿。

**通过标准**

- blocker 来自客观外部条件、阶段授权、资格、密钥、破坏性决策或用户待拍板事项。
- 运行状态保持 `BLOCKED / FAIL_EXTERNAL / DEFERRED` 等真实状态。
- defer 有权威依据、owner、任务 ID 或等价可追踪条目。
- mock、replay、stub、sandbox 或 synthetic PASS 只可作为低档验证，不得替代 blocker。

**典型 FAIL**

- 缺密钥时切到 mock，仍把同一行写成 live PASS。
- 上游不可达时用 replay 刷绿，未保留外部失败状态。
- 缺牌照、终端或资格时用占位探针冒充完成。
- 未纳入本阶段的能力没有书面 defer，却被口头视为完成。

**证据最低要求**

真实 blocker 状态与原因；权威/决议引用；owner；任务 ID 或等价条目；证明未使用替身覆盖 blocker 的证据。

**NA 边界**

对象没有外部 blocker、阶段 defer 或资格限制时可 `NA`。

## 5. 主分类边界

主分类只用于给发现归档；七行评估仍全部执行。

### 2 / 5 / 6 / 7

| 先问                                       | 主分类 |
| ------------------------------------------ | ------ |
| 权威承诺的能力根本没有真实落地？           | `CC-2` |
| 已有交付，但步骤、字段或形状与权威不一致？ | `CC-5` |
| 本轮做得到的清理或共享根因修复被推迟？     | `CC-6` |
| 客观做不了，且 defer 有依据与追踪项？      | `CC-7` |

### 1 / 3 / 4 与替身

| 观察事实                                     | 主分类 |
| -------------------------------------------- | ------ |
| 测试全绿但没有业务 outcome 证明              | `CC-1` |
| 测试专属接线与正式入口不同                   | `CC-3` |
| replay/mock/sandbox 证据被标成更高档位       | `CC-4` |
| stub/mock 说明真实能力未实现                 | `CC-2` |
| mock/replay 用来掩盖本应保持的 blocker/defer | `CC-7` |

同一事件允许出现多个 FAIL，例如“测试注入后 replay 绿并声称 live”可同时触发 `CC-1`、`CC-3`、`CC-4`。

## 6. 关账检查

一个对象只有同时满足以下条件才可 `CLOSED`：

1. 对象、完成声称和权威明确。
2. `CC-1` 至 `CC-7` 七行齐全且顺序稳定。
3. 无 `FAIL`；每个 `PASS` 有运行事实与复现证据；每个 `NA` 有客观原因。
4. `CC-5 PASS` 已给出权威形、实际形与偏差结论。
5. `CC-6 PASS/NA` 未掩盖本轮能修未修的事项。
6. `CC-7 PASS` 若确认有效 defer，则对象状态为 `BLOCKED` 而非 `CLOSED`。
7. Summary 与逐行判定一致，不得用整体“测试全绿”覆盖单行缺口。

## 7. 维护规则

- 改分类语义、ID、代号或拷问：提升核心版本，并同步 `CONTRACT.md` 与 `EVALS.md`。
- 新增项目案例：写入项目自己的 `completion-check.project.md`，不要直接堆入本文件。
- 只有跨项目稳定复现、能改善边界判断的模式，才可蒸馏为本文件的通用示例。
- 删除重复、项目词汇、时效性流程和消费端编排；保持本文件只承载语义核心。
