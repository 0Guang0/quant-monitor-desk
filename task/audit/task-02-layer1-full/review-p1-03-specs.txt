### Top Issues

1. **CI validation can fail unclearly or misleadingly for malformed registry input**
   `scripts/check_indicator_binding_matrix.py` lets unreadable or malformed YAML escape as an exception before producing the structured validation errors expected in CI. The registry load path should catch `read_text` / `yaml.safe_load` failures and report a validation error that includes the registry path and underlying parse/read issue.

2. **Registry schema validation is too weak, causing bad input to surface as parity failures**
   `scripts/check_indicator_binding_matrix.py` normalizes `payload.get("bindings") or []`, which hides missing, null, or wrong-type `bindings` values as an empty list. It also checks required fields only by key presence, allowing empty strings, wrong scalar types, stringified booleans, and invalid list contents to pass into later validation. This makes CI diagnostics point at row-count or parity mismatches instead of the actual schema defect.

3. **Axis spec discovery failures are reported as registry content errors**
   In `scripts/check_indicator_binding_matrix.py`, a missing axis spec directory or glob/layout change can produce an empty `axis_ids` set, after which every registry row is reported as unknown. CI should distinguish ‚Äúno specs discovered‚Äù from ‚Äúregistry references unknown axis IDs.‚Äù

4. **Write-capable CLI contract entries are missing safety defaults/guards**
   `specs/contracts/data_cli_contract.yaml` has write-capable commands with weaker safety contracts than peers. `qmd data sync` lacks `default_mode: dry_run`, and `reconcile` appears mutating but does not require `ResourceGuard`. These omissions can allow wrappers or implementations to take live-write paths without the shared safety/confirmation mechanism.

### Module Hotspots

- **Indicator binding validation script:** `scripts/check_indicator_binding_matrix.py` has most of the findings. The issues cluster around input parsing, schema validation, and discovery diagnostics, all affecting CI signal quality.
- **Data CLI contract:** `specs/contracts/data_cli_contract.yaml` has fewer comments but higher operational risk because missing contract safety requirements can affect write-capable command behavior.

### Cross-Cutting Concerns

- **Validation reports secondary symptoms instead of root causes**
  Representative path: `scripts/check_indicator_binding_matrix.py`. Bad YAML, wrong `bindings` shape, invalid field types, and failed spec discovery can all produce exceptions or misleading parity/unknown-ID errors instead of precise validation diagnostics.

- **Machine-readable contracts do not consistently encode safety invariants**
  Representative path: `specs/contracts/data_cli_contract.yaml`. Mutating commands should consistently declare dry-run defaults and required guardrails such as `ResourceGuard`.

### Quick Wins

- Wrap registry file reading and YAML parsing in `scripts/check_indicator_binding_matrix.py` and emit structured validation errors with the registry path and exception details.
- Preserve the raw `bindings` value, explicitly require the key to exist, and validate that it is a list before row-count/parity checks.
- Add strict field validation for registry rows: non-empty strings for IDs, real booleans for boolean fields, and correctly typed list contents.
- Add an explicit discovery precheck for the axis spec directory and for zero matched/discovered spec IDs.
- Update `specs/contracts/data_cli_contract.yaml` so `qmd data sync` declares `default_mode: dry_run`.
- Require `ResourceGuard` for `reconcile` if it mutates conflict state.
