**Data-loss assessment:** Migration 006 only creates empty tables and indexes — no row migration or destructive DDL. Kill-and-rerun cannot corrupt existing ingestion data in 004/005 tables (`source_registry`, `fetch_log`, `validation_report`, etc.) because 006 does not `ALTER` or `DROP` them. Runtime job/event rows written after migration are unaffected by re-running `init_db` (idempotent skip). **Residual risk (inherent, not Batch D–specific):** if a migration file is edited after apply, `verify_applied_checksums` (`migrate.py:31-51`) correctly raises `MigrationChecksumError` — operators must not rewrite applied migrations.
